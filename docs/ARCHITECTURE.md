# 아키텍처

## 디렉토리 구조
```
src/
├── app/               # 메인 페이지 + API 라우트
├── components/        # 업로드, 리스트, 상태, 액션 UI 컴포넌트
├── types/             # TypeScript 타입 정의
├── lib/               # PDF 검증, 병합 코어, 파일명 생성, 정리 로직
└── cli/               # 로컬 CLI 엔트리포인트와 인자 파서
```

## 패턴
- Next.js App Router 기반 단일 애플리케이션으로 웹 UI와 API를 함께 제공한다.
- PDF 병합 로직은 `lib/`의 공용 서비스로 분리하고, 웹 API와 CLI가 같은 로직을 호출한다.
- 클라이언트는 파일 선택, 순서 조정, 상태 표시만 담당하고 실제 병합은 서버에서 수행한다.
- PDF 처리 모듈은 클라이언트 번들에 포함하지 않는다.
- 병합 대상은 PDF 페이지 내용이며, 북마크/폼/첨부파일/원본 메타데이터 보존은 목표로 삼지 않는다.

## 데이터 흐름
```
WEB
사용자 파일 선택
→ 클라이언트 1차 검증 (PDF 여부, 개수, 용량)
→ 파일 리스트 생성
→ 동일 파일명 표시명 보정 (예: report.pdf, report (2).pdf)
→ 드래그 또는 위/아래 버튼으로 순서 조정
→ POST /api/merge 로 정렬된 파일 전송
→ 서버 2차 검증
→ 메모리 우선 병합, 필요 시 임시 디스크 사용
→ 병합 결과 PDF 생성
→ application/pdf 바이너리 응답 반환
→ 응답 완료/실패/취소 후 임시 자원 정리

CLI
사용자 명령 입력
→ 파일 경로 존재 여부 및 형식 검증
→ 공용 병합 서비스 호출
→ 출력 파일 경로 계산
→ 동일 경로 존재 시 suffix 기반 새 이름 생성
→ 병합 결과 저장
→ 성공 또는 오류 코드 반환
```

## 상태 관리
- 메인 UI 상태는 `empty`, `ready`, `running`, `success`, `error` 다섯 가지로 관리한다.
- 사용자에게 노출하는 진행 단계는 `파일 준비 완료`, `업로드/검증 중`, `병합 중`, `다운로드 준비 완료` 네 단계로 고정한다.
- 파일 리스트 항목은 최소 `id`, `displayName`, `originalName`, `size`, `order`, `status`, `sourceFile` 정보를 가진다.
- 병합 중에는 업로드, 삭제, 재정렬, 병합 재실행을 잠그고 `취소`만 허용한다.
- 취소는 클라이언트 요청 취소 기준이며, 서버는 연결 종료를 감지하면 가능한 범위에서 중단하고 정리한다.
- 서버는 요청 간 상태를 저장하지 않는 무상태 구조를 유지한다.

- 입력 제한:
  - 최소 2개 파일부터 병합 가능
  - 최대 20개 파일
  - 파일당 최대 50MB
  - 총합 최대 200MB
- API 인터페이스:
  - `POST /api/merge`
  - 성공 시 `200` + `application/pdf` + `Content-Disposition`
  - 검증 실패 시 `400` JSON 오류
  - 서버 병합 실패 시 `500` JSON 오류
- CLI 인터페이스:
  - `pdf-merge <input...> [-o output.pdf]`
  - 입력 인자 순서가 병합 순서
  - `-o` 미지정 시 `merged-YYYYMMDD-HHMMSS.pdf` 규칙 적용
  - 출력 파일 충돌 시 자동으로 새 이름 생성
