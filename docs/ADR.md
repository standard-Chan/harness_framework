# Architecture Decision Records

## 철학
MVP 범위를 엄격하게 지키고, 웹과 CLI가 동일한 핵심 로직을 공유하며, 사용자 입장에서 한 번에 이해되는 단순한 흐름을 우선한다.

---

### ADR-001: Next.js App Router 기반 단일 애플리케이션 선택
**결정**: 웹 UI와 API를 Next.js App Router 기반 단일 애플리케이션으로 구현한다.
**이유**: UI와 API를 같은 코드베이스에서 관리하면 개발과 배포가 단순해지고, Harness step 분리 시에도 맥락 유지가 쉽다.
**트레이드오프**: 순수 CLI 또는 백엔드 전용 구조보다 Node 런타임 의존성이 커진다.

### ADR-002: TypeScript strict mode 채택
**결정**: 웹 UI, API, 공용 병합 로직, CLI를 TypeScript strict mode 기준으로 설계한다.
**이유**: 파일 상태, API 응답, CLI 인자, 병합 옵션을 타입으로 고정하면 구현 단계에서 추상적 판단을 줄일 수 있다.
**트레이드오프**: 초기 타입 설계 비용이 늘어난다.

### ADR-003: pdf-lib 기반 공용 병합 코어 채택
**결정**: PDF 병합 엔진은 `pdf-lib`를 사용하고, 웹 API와 CLI가 동일한 공용 코어를 호출한다.
**이유**: 외부 바이너리 의존 없이 TypeScript 친화적으로 구현할 수 있고, 웹과 CLI 간 결과 일관성을 유지하기 쉽다.
**트레이드오프**: 고급 PDF 복구나 복잡한 PDF 자산 보존 기능은 지원 범위 밖이 된다.

### ADR-004: 병합은 서버에서 수행
**결정**: 브라우저가 아니라 서버에서 PDF 병합을 수행한다.
**이유**: 브라우저 메모리 부담과 라이브러리 제약을 줄이고, 대용량 파일 처리와 결과 일관성을 확보할 수 있다.
**트레이드오프**: 업로드 비용과 서버 자원 사용량이 발생한다.

### ADR-005: 동기 POST /api/merge 응답으로 직접 다운로드
**결정**: 웹 병합은 `POST /api/merge` 한 요청 안에서 처리하고, 성공 시 병합된 PDF 바이너리를 직접 반환한다.
**이유**: MVP 구조가 단순해지고, 사용자는 별도 작업 조회 없이 바로 다운로드를 받을 수 있다.
**트레이드오프**: 장시간 작업 큐나 상태 조회 기반 비동기 처리 구조는 제공하지 않는다.

### ADR-006: 메모리 우선 + 필요 시 임시 디스크 사용
**결정**: 병합 처리 시 메모리 사용을 우선하되, 서버 안정성을 위해 필요 시 임시 디스크 사용을 허용한다.
**이유**: 작은 파일은 빠르게 처리하고, 동시에 큰 파일 입력에서도 서버가 불안정해지는 것을 줄일 수 있다.
**트레이드오프**: 메모리 전용 구조보다 구현 복잡도가 조금 증가한다.

### ADR-007: 임시 파일과 결과 파일은 작업 종료 즉시 삭제
**결정**: 업로드 파일과 결과 파일은 성공, 실패, 취소 후 즉시 정리한다.
**이유**: 무인증 MVP에서 개인정보와 운영 부담을 줄이고, 저장소 관리 없이 단순한 구조를 유지할 수 있다.
**트레이드오프**: 재다운로드 이력이나 작업 보관 기능을 제공할 수 없다.

### ADR-008: 애플리케이션 레벨 동시성 제한은 두지 않음
**결정**: 전역 작업 큐나 동시 병합 개수 제한을 애플리케이션에 넣지 않고, 각 요청을 독립 처리한다.
**이유**: MVP 복잡도를 낮추고, 요청 흐름을 단순하게 유지할 수 있다.
**트레이드오프**: 서버 리소스 보호는 인프라 운영과 배포 환경 튜닝에 더 의존하게 된다.

### ADR-009: 취소는 클라이언트 요청 취소 기준으로 정의
**결정**: 웹의 취소 기능은 브라우저 요청 취소를 의미하며, 서버는 연결 종료를 감지하면 가능한 범위에서 중단 및 정리를 시도한다.
**이유**: 사용자 경험상 즉시 멈출 수 있는 조작을 제공하면서도, 서버에서 과도한 작업 제어 복잡도를 피할 수 있다.
**트레이드오프**: 서버 작업의 완전한 즉시 중단을 보장하지는 않는다.

### ADR-010: 결과 PDF는 페이지 병합 정확성만 보장
**결정**: 결과 PDF는 입력 순서에 따른 페이지 병합 정확성만 보장하고, 북마크, 폼, 첨부파일, 메타데이터 보존은 목표로 삼지 않는다.
**이유**: MVP에서 핵심 가치를 페이지 순서대로 하나의 PDF를 만드는 것으로 한정하면 구현과 테스트 범위를 안정적으로 유지할 수 있다.
**트레이드오프**: 고급 PDF 워크플로우 사용자에게는 기능이 부족할 수 있다.

### ADR-011: 라이트 모드 고정
**결정**: UI는 라이트 모드만 지원한다.
**이유**: 일반 사용자에게 익숙한 문서 작업 환경을 제공하고, 상태 대비와 인쇄물 친화적 인상을 주기 쉽다.
**트레이드오프**: 다크모드 선호 사용자는 별도 테마를 사용할 수 없다.

### ADR-012: 친근한 소비자 앱형 도구 톤 선택
**결정**: 관리 콘솔형이 아니라 친근한 소비자 앱형 생산성 도구 톤으로 설계한다.
**이유**: 단발성 사용자가 첫 화면에서 바로 조작을 이해하고 심리적 진입 장벽을 낮출 수 있다.
**트레이드오프**: 전통적인 기업용 관리자 UI보다 덜 엄격하게 보일 수 있다.

### ADR-013: 스텝 바 + 상태 카드 조합 사용
**결정**: 진행 상태는 단일 스피너가 아니라 스텝 바와 상태 카드 조합으로 표현한다.
**이유**: 현재 단계와 다음 가능한 행동을 동시에 보여주기 쉬워 사용자가 멈춤 상태를 오해하지 않는다.
**트레이드오프**: 단순 텍스트 상태보다 화면 면적을 더 사용한다.

### ADR-014: 모바일 1열 작업 흐름 채택
**결정**: 모바일에서는 업로드, 리스트, 상태, 액션, 결과를 1열로 재배치한다.
**이유**: 터치 환경에서 위아래 이동 버튼과 상태 확인을 더 단순하게 구성할 수 있다.
**트레이드오프**: 데스크톱보다 한 번에 보여줄 수 있는 정보량은 줄어든다.
